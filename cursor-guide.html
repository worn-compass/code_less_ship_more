00 — OVERVIEW

What Cursor actually is

And why it's different from just using ChatGPT for code

Cursor is a code editor (built on VS Code) with AI baked into everything. The key difference from other AI tools: it understands your entire project, not just a snippet you paste in.

🧠 Context-aware AI

Cursor reads your whole codebase. Ask "how does auth work in this app?" and it'll give you a real answer based on your actual code — not a generic tutorial.

⚡ Inline generation

Select any file, press a key, describe what you want. The AI edits code directly — you review the diff and accept or reject like a PR.

🔍 Explain anything

Highlight confusing code, ask "what does this do?" — get a plain-English answer. No more Googling cryptic function names.

🚀 Multi-file edits

Tell it to "add a new settings page with these fields" and it'll touch every file that needs changing — routes, components, types, all of it.

💡 Your unfair advantage as a PM

You don't need to ship production code. You need to prototype fast enough to test ideas, understand what devs are telling you, and produce handoffs that aren't vague. Cursor is purpose-built for exactly this.

01 — SETUP

Getting set up

From download to your first AI edit in under 15 minutes

01

Download and install

Go to cursor.com and download for your OS. It installs like any app. Sign in with your work account — your license should auto-apply.

02

Open a project folder

Use File → Open Folder (not just a file). This is critical — Cursor's AI works best when it can see the whole project. If you're exploring an existing codebase, clone it from GitHub first, then open the folder.

03

Pick your AI model

Cursor lets you choose which AI model to use. For most things, Claude Sonnet or GPT-4o are excellent. You can switch models per-conversation in the chat panel. Start with the default — you can experiment later.

04

Index your codebase

Go to Cursor Settings → Features → Codebase Indexing and enable it. Cursor will scan your project so the AI can reference any file, even ones you haven't opened. This takes a few minutes for large codebases.

05

Try your first AI edit

Open any file. Press Cmd+K (Mac) or Ctrl+K (Windows). Type "add a comment explaining what this function does." Hit Enter. Accept the change. You're now using Cursor.

Pro tip Import your VS Code settings if you've used it before: Cursor → Import VS Code Settings. Your extensions, themes, and keybindings carry over instantly.
02 — INTERFACE

The Interface

The 5 areas you actually need to know

📁 Explorer (left sidebar)

Your file tree. Click files to open them. Right-click for options. The search bar at top (Cmd+P) is your real navigation — type a filename and jump to it instantly. Much faster than clicking through folders.

✏️ Editor (center)

Where code lives. Click to place your cursor, then hit Cmd+K to trigger inline AI. You'll see AI-suggested edits highlighted in green (adds) and red (removes) — like a code diff. Press Tab to accept, Escape to reject.

💬 Chat Panel (right, or Cmd+L)

Your main AI interface. Type questions about the codebase, ask it to write code, debug errors, or explain concepts. Use @filename to pull in specific files as context. This is where you'll spend a lot of time.

🔮 Composer (Cmd+I)

The power tool for bigger changes. Unlike Chat which mostly advises, Composer directly edits multiple files at once. Tell it "add a dark mode toggle to the settings page" and it'll touch every file needed.

⚙️ Terminal (bottom, Ctrl+`)

A standard command-line terminal built in. Useful for running your app locally, installing packages, or using git. Cursor's AI can also suggest terminal commands when you ask in Chat.

03 — AI TOOLS

Chat & Composer

The two ways to talk to AI in Cursor — and when to use each

Chat (Cmd+L)

Conversational. Ask questions, get explanations, request code snippets. Great for exploration and understanding. AI responds in chat, you copy/paste or apply manually.

Explain code Ask questions Debugging
Composer (Cmd+I)

Agentic. Give it a task and it edits files directly across your project. Best for building new features, prototyping, or making changes that span multiple files.

Build features Multi-file edits Prototyping

The @ symbol — your context superpower

In both Chat and Composer, typing @ opens a context menu. This lets you pull in exactly what the AI should know about.

  • @filename — Attach a specific file to the conversation
  • @folder — Include an entire folder of files
  • @codebase — Let AI search the full indexed codebase
  • @docs — Pull in official library documentation (React, Next.js, etc.)
  • @web — Live web search for current info
  • @git — Reference recent commits and diffs
Practical example You're trying to understand why a bug exists. Open Chat and type: "@codebase Why does the user session expire early? Trace through the auth flow and explain it in plain English." Cursor will read the relevant files and explain it like a good engineer would.
04 — CODEBASE EXPLORATION

Reading codebases

Use Cursor to get up to speed on any project fast — even ones you didn't build

🎯 PM use case

Before any sprint planning, backlog grooming, or technical spec — spend 20 minutes with Cursor asking questions about the codebase. You'll walk into the meeting with better context than most engineers who haven't touched that area recently.

Questions to ask Cursor about any codebase

@codebase Give me a high-level overview of how this app is structured. What are the main modules and how do they relate?
@codebase How does [feature name] work end-to-end? Walk me through it from the user action to the database.
@codebase What would need to change if we wanted to add [new feature]? List the files and areas that would be affected.
@codebase What are the main data models? Describe each one and what it represents in the product.
@codebase Where are the biggest areas of technical debt or known fragility? What should I be careful about?

The "explain this" trick

1

Highlight confusing code

Select any block of code in the editor that you don't understand.

2

Right-click → "Add to Chat"

Or press Cmd+Shift+L. The selected code gets sent to the chat panel.

3

Ask your question

"Explain this to a non-engineer" or "what problem is this solving?" or "why would someone write it this way?"

Gotcha If you ask about a function and Cursor seems confused, it probably needs more context. Add @codebase to your question so it can look at how that function is used elsewhere.
05 — PROTOTYPING

Prototyping fast

From idea to working demo — the PM's secret weapon

🎯 The goal

You're not trying to ship production code. You're trying to build something real enough to get meaningful feedback from users, stakeholders, or your own engineering team. Think of it as an interactive Figma that actually works.

The prototype playbook

01

Start with a scaffold prompt

Open Composer (Cmd+I) in a new or empty folder. Give it a clear, specific task to build the skeleton of what you need.

Build a simple React app with a sidebar navigation, a main content area, and a settings page. Use Tailwind for styling. Keep it clean and minimal.
02

Iterate with plain English

Don't try to write perfect specs upfront. Build fast, react to what you see. Keep Composer open and keep telling it what to change.

Add a user profile card in the sidebar with an avatar, name, and role. Make it look polished.
03

Use real-ish data

Ask Cursor to generate placeholder data that looks realistic — real names, real-looking IDs, plausible numbers. This makes demos far more convincing.

Replace the mock data with 5 realistic user records. Use realistic names, email addresses, and dates.
04

Run it locally

Ask Cursor in Chat: "How do I run this app?" It'll tell you exactly what to type in the terminal. Usually just npm install then npm run dev.

05

Document what you built

Ask Cursor: "Write a brief technical summary of this prototype — what it does, what's hardcoded, and what would need to change for production." Paste this into your handoff doc.

Great prototype stacks for PMs React + Tailwind — Fast UI, easy to tweak visually
Next.js — If you need pages, routing, or mock API endpoints
Plain HTML/CSS/JS — When you want maximum simplicity and shareability

07 — BEST PRACTICES

Best practices

How to actually get good results, not just fast results

📐 Always open a folder, not a file

Cursor is dramatically more useful when it can see the whole project. Opening a single file cuts the AI off from context it needs. Make it a habit: always Open Folder.

🔁 Small, focused prompts beat big vague ones

Instead of "build me a dashboard," say "add a chart component to the analytics page that shows weekly signups using recharts." Specific = better output, fewer iterations.

✅ Review every AI edit before accepting

Cursor shows you diffs in green/red. Don't just Tab through everything. A 10-second review catches 90% of AI mistakes. Treat it like a junior dev's PR — smart but needs checking.

💾 Commit often with Git

If you break something, you need a way back. Even for prototypes, run git init and commit after each working state. Ask Cursor in Chat "what git commands should I run to save my progress?"

🧩 Use .cursorrules to set defaults

Create a file called .cursorrules in your project root. Write plain English rules like "always use TypeScript," "prefer functional components," or "never use inline styles." The AI will follow these automatically.

# .cursorrules example Always use TypeScript. Use Tailwind for styling, not inline CSS. Write plain English comments for any non-obvious logic. Keep components small and focused — one responsibility each.
🔗 Give the AI its own error messages

If something breaks, paste the error message directly into Chat. Don't try to interpret it first. Type: "I got this error: [paste]. What does it mean and how do I fix it?" Nine times out of ten, you'll have an answer in under a minute.

08 — PROMPTING

Prompting like a pro

The difference between mediocre and great AI output is usually the prompt

The anatomy of a good Cursor prompt

1. Role / context — Tell it who you are or what the project is

2. Task — Be specific about what you want built or changed

3. Constraints — Style, libraries, what NOT to do

4. Output format — "Explain it," "edit the file," "write a new component," etc.

This is a Next.js app using Tailwind. I need to add a notification banner that appears at the top of every page when there's a system alert. It should be dismissable, use amber/yellow for warnings and red for errors, and store the dismissed state in localStorage. Edit the Layout component to add this.

Prompts for your specific PM workflow

Understanding existing code
@codebase Explain the [feature] flow in plain English. Who calls what, what data changes, and what does the user see?
@filename What is this file responsible for? What would break if I deleted it?
Estimating effort for stakeholders
@codebase If we wanted to add [feature], list the files that would need to change and give a rough sense of complexity: trivial / moderate / significant.
Prototype scaffolding
Build a [type] prototype. It should: - [Key screen 1] - [Key screen 2] - [Core interaction] Use hardcoded data for now. Make it look polished enough to demo.
Writing documentation
@codebase Write a technical summary of this project for a new engineer joining the team. Cover architecture, main data models, and anything they should know before making changes.
09 — HANDOFF

Handing off to devs

How to use Cursor-built prototypes to have better engineering conversations

🎯 The goal of handoff

A Cursor prototype is not production code. It's a communication tool — it shows devs exactly what you mean so they spend less time guessing and more time building the right thing.

The handoff checklist

  • Screenshot or record the prototype working. A 2-minute Loom beats a 5-page spec.
  • List what's hardcoded vs. what should be real. Ask Cursor to generate this list.
  • Document your data model assumptions — what fields you used, what relationships you implied.
  • Flag edge cases you didn't handle — error states, empty states, loading states.
  • Note which libraries you used — devs may want to swap them for internal standards.
  • Share the repo — even messy prototype code is better than nothing for engineers to reference.

Let Cursor write the handoff doc

@codebase Write a technical handoff document for this prototype. Include: - What it does (user-facing) - How it's structured (technical overview) - What's hardcoded vs. needs real implementation - Assumptions I made about data models - Edge cases not handled - Suggested next steps for a dev picking this up
The real win When you walk into the engineering meeting and say "here's a working prototype, here's a technical summary, and here's what needs to change for production" — the conversation gets 10x more productive. You've done their first hour of work for them.
10 — WATCH OUT

Traps to avoid

Things that trip up even experienced Cursor users

⚠️ Trusting AI output blindly

Cursor can confidently write code that doesn't work, or that works for the happy path but breaks on edge cases. Always test what it builds, even briefly. The AI is wrong just often enough to matter.

⚠️ Letting context get stale

If you've made a lot of changes and the AI starts giving weird suggestions, start a fresh Chat or Composer session. Long conversations can confuse the model's understanding of the current state of your code.

⚠️ Not using @codebase when you should

If the AI gives a generic answer that ignores your actual project, it probably doesn't have context. Add @codebase to your question so it searches your indexed code first.

⚠️ Huge prompts for huge tasks

Asking Cursor to "build the entire onboarding flow" in one go usually produces mediocre results. Break big tasks into smaller steps. Each Composer prompt should be one coherent change.

⚠️ Showing prototypes as finished work

Cursor-built prototypes skip auth, error handling, accessibility, performance, security, and a dozen other things that matter in production. Always label your prototypes clearly as "not production-ready."

⚠️ Forgetting to save / commit

AI edits can cascade in unexpected ways. Get in the habit of committing to git after each working state. If you don't know git, ask Cursor to walk you through it — it'll teach you the 5 commands you actually need.

11 — POWER MOVES

Hidden power moves

Things most new Cursor users don't discover for months

🗺️ Generate an architecture diagram underused

Ask: "@codebase Generate a Mermaid diagram showing how the main components of this app relate to each other." Paste the output into mermaid.live and you get a visual map. Great for onboarding docs and engineering discussions.

🐛 Paste errors directly underused

When something breaks in your browser or terminal, copy the full error message and paste it in Chat. No preamble needed. Cursor knows what to do with a stack trace.

🧪 Ask for test scenarios pm gold

Ask: "@codebase What are the 10 most important test cases for the checkout flow? List happy path, edge cases, and potential failure modes." This is legitimately useful for QA planning, not just engineering.

📊 Data model reverse engineering pm gold

Ask: "@codebase Describe the data model of this app. What are the main entities, their fields, and how they relate? Format it like an ERD description." Understand what's being stored before you spec new features.

📝 Auto-generate user stories new habit

After building a prototype, ask: "Looking at this code, write user stories in the format 'As a [user], I want to [action] so that [outcome]' for all the features I've built." Instant backlog seed.

🔐 Security and privacy audit new habit

Before showing any prototype to stakeholders, ask: "@codebase Look at this code and flag any obvious security issues, hardcoded credentials, or privacy concerns I should be aware of." Good habit even for prototypes.

🌐 Multi-model comparison

If Cursor gives a confusing answer, switch models in the chat dropdown (Claude vs GPT-4o) and ask the same question. Different models have different strengths. Claude tends to be better at long explanations; GPT-4o sometimes writes tighter code.

📦 Notepads for persistent context

Use Cursor Notepads (under Chat) to save reusable context you add to every session — like your tech stack, coding standards, or product description. Reference them with @notepad in any chat.

The bigger picture Cursor makes you a more informed PM. When you can read code, prototype ideas, and understand technical tradeoffs — your specs get tighter, your estimates get more accurate, and your engineers spend more time building and less time clarifying. That's the real ROI.
12 — AGENT RULES

Setting rules for the agent

How to stop re-explaining yourself every session and make the AI behave consistently

Without rules, Cursor's agent starts fresh every session — no memory of your stack, your preferences, or your standards. Rules fix that. They're instructions that run silently in the background on every prompt.

🎯 Why this matters for you

As a PM prototyping across multiple projects, rules let you clone your "working style" into every new repo. You write them once, and the agent behaves like a dev who's already been onboarded — knows your stack, your naming conventions, what to skip, what to never do.

The two places rules live

.cursorrules project-level

A plain text file in your project root. Rules here apply only to this project. Committed to git — your whole team inherits them. Best for stack-specific, project-specific instructions.

Settings → Rules for AI global

Lives in Cursor's settings, applies across every project. Best for personal preferences that never change — tone, output format, how much the AI explains its reasoning.

What to put in your rules

Stack & language rules

Tell it exactly what you're using so it never suggests the wrong thing.

This is a Next.js 14 app using the App Router (not Pages Router). TypeScript only — never suggest plain JavaScript. Use Tailwind for all styling. No CSS modules, no styled-components. State management via Zustand. Do not suggest Redux or Context API. Use Prisma for database queries. PostgreSQL.
Code style & conventions

Enforce consistency without having to say it every time.

Use named exports, not default exports. Component files: PascalCase. Utility files: camelCase. Keep components under 150 lines. If longer, split them. Always handle loading and error states in UI components. No console.log statements in committed code — use a logger.
Behavior & output style

Control how the agent communicates and what it assumes.

When making changes, briefly explain what you changed and why. If you're unsure about something, say so — don't guess silently. Don't refactor code I didn't ask you to touch. When adding new dependencies, flag them explicitly. Prefer editing existing files over creating new ones unless necessary.
PM-specific prototype rules

A starter .cursorrules for quick PM prototyping work.

# PM Prototype Rules This is a prototype — not production code. Prioritize clarity and speed over optimization. Use hardcoded mock data unless told otherwise. Keep all components in /components. Pages in /app or /pages. Write simple, readable code — I'll be sharing this with engineers who need to understand it quickly. Add a short comment above any non-obvious logic. If a feature would require auth, backend, or a real API to work, stub it out and add a TODO comment explaining what's needed.
Start simple, add over time Don't try to write the perfect .cursorrules on day one. Start with your stack and one or two style preferences. Add rules whenever you catch the agent doing something you had to correct — that correction is your next rule.

Rules don'ts

  • Don't write rules as vague principles. "Write good code" does nothing. "Keep functions under 30 lines" does something.
  • Don't over-specify things the AI already knows. You don't need "use proper indentation" — that's noise that dilutes real rules.
  • Don't set conflicting rules. "Always use TypeScript" and "write plain .js files" will confuse the agent. One wins arbitrarily.
  • Don't forget to update them. If your stack changes, update the rules. Stale rules cause confusing AI behavior that's hard to debug.
13 — DIRECTORY SETUP

Directory setup for AI projects

Structure your project so the agent always knows where things go — and where to look

A well-structured directory does two things: it makes the AI's outputs predictable (new files land in the right place), and it makes codebase exploration dramatically easier for you and your devs.

The principle AI agents work best with opinionated, predictable structure. Flat folders with 40 files, or deeply nested trees with no pattern, both degrade output quality. Clear conventions = better AI, better handoffs, better everything.

Recommended structure for PM prototypes (Next.js)

/my-prototype ├── /app ← Pages and routes (Next.js App Router) │ ├── /dashboard │ │ └── page.tsx │ ├── /settings │ │ └── page.tsx │ └── layout.tsx ├── /components ← Reusable UI components │ ├── /ui ← Generic: Button, Card, Modal, Input │ └── /features ← Feature-specific: UserCard, MetricsChart ├── /lib ← Utilities, helpers, constants │ ├── utils.ts │ └── constants.ts ├── /data ← Mock/hardcoded data lives here │ └── mockUsers.ts ├── /types ← TypeScript types and interfaces │ └── index.ts ├── /public ← Static assets (images, icons) ├── .cursorrules ← Your agent rules ├── PLANNING.md ← Your project brief └── TASKS.md ← Current task list for the agent

The two files that change everything: PLANNING.md and TASKS.md

PLANNING.md

A living document at the root of your project. Describes what you're building, why, the tech stack, key decisions, and constraints. Reference with @PLANNING.md. Think of it as the onboarding doc for your AI collaborator.

TASKS.md

A running list of what needs to be done. You update it as work progresses. Reference it at the start of each Composer session. Keeps the agent focused and gives you a natural checklist as PM.

Sample PLANNING.md
# Project: [Name] ## What this is One paragraph. What does this app do? Who uses it? What problem does it solve? ## Tech stack - Framework: Next.js 14 (App Router) - Styling: Tailwind CSS - Language: TypeScript - Data: Hardcoded mock data (no real backend yet) ## Key screens 1. Dashboard — overview of [X] 2. Detail view — drill into [Y] 3. Settings — manage [Z] ## What this is NOT - Not production-ready - No real auth - No real API calls ## Key decisions - Using Recharts for charts (not D3 — too complex for prototype) - All mock data lives in /data folder - Mobile responsive but desktop-first
Sample TASKS.md
# Tasks ## In Progress - [ ] Add empty state to the user table when no results ## Up Next - [ ] Build the notifications dropdown in the header - [ ] Add date range filter to the dashboard chart - [ ] Make sidebar collapsible on desktop ## Done - [x] Scaffold app shell and navigation - [x] Build dashboard page with mock metrics - [x] User table with sorting
How to use these together Start every Composer session with: @PLANNING.md @TASKS.md — let's work on the next task in the list. The agent has full context before writing a single line of code.
14 — PLAN VS. BUILD

Plan vs. Build mode

The most important strategic distinction in AI-assisted development

Cursor's agent can do two very different things: think with you (Plan) or execute for you (Build). Knowing which mode you're in — and switching intentionally — is what separates people who get great results from people who get frustrated.

🗺 Plan Mode

You're thinking, designing, deciding. The AI is a thinking partner — it challenges assumptions, identifies gaps, proposes approaches, and helps you figure out what to build before building it.

Tools: Chat. Conversation.

Output: Decisions, PLANNING.md, TASKS.md, clarity.

🔨 Build Mode

You know what you want. The AI executes. It writes code, edits files, creates components. You review diffs, test what it builds, and steer with follow-up prompts.

Tools: Composer. Inline edit (Cmd+K).

Output: Working code, running prototype.

The most common mistake Jumping straight to Build mode before the plan is solid. The AI will confidently build something — it just might not be the right thing. Vague prompt in = plausible-looking but wrong code out. You'll spend more time correcting it than if you'd planned for 20 minutes first.

When is your plan good enough to build?

The "ready to build" checklist
  • You can name every screen. If you can't list the pages, you're not ready.
  • You know what data is on each screen. Even roughly — what does the user see? What are the key fields?
  • You've chosen your stack. Framework, styling, key libraries. Don't let the agent choose for you unless you genuinely don't care.
  • You know the first task. Not the whole project — just the first concrete thing to build. "Scaffold the app shell with navigation" is a perfect first task.
  • You know what you're NOT building yet. Explicitly listing out-of-scope items prevents the agent from over-building.

The planning conversation prompt

I'm a PM building a prototype for [describe the idea in 2-3 sentences]. The users are [who]. The core problem it solves is [problem]. Help me think through: 1. What are the key screens this prototype needs? 2. What data/fields does each screen need? 3. What's a good tech stack for a quick React prototype? 4. What should I explicitly leave out for now? 5. What's a logical order to build the screens? Then write me a PLANNING.md and TASKS.md I can add to my project.
The 20-minute rule If you can't describe what you're building clearly enough to write a PLANNING.md in 20 minutes of back-and-forth with the AI, you need more product thinking — not more coding. That's a product problem, not a Cursor problem.
15 — PROJECT ARTIFACTS

Project artifacts

What documents and files to expect — and create — across any project

A "project artifact" is any file that describes, plans, or documents your project rather than running it. These are just as important as the code — especially when handing off to devs or returning to a prototype weeks later.

📋 PLANNING.md essential

What the project is, why it exists, the tech stack, key decisions, and what's out of scope. Lives at project root. Start every project with this.

✅ TASKS.md essential

Current work queue: In Progress, Up Next, Done. The agent uses this as its work order. You use it as a PM dashboard. Update as you go — it doubles as a changelog.

📐 ARCHITECTURE.md for handoff

How the app is structured technically. Component hierarchy, data flow, key wiring decisions. Ask Cursor to generate it: @codebase Write an ARCHITECTURE.md for this project.

🗄️ DATA_MODEL.md for handoff

Every data entity, its fields, types, and relationships. Even for mock data — tells devs what the real schema should look like. Prompt: @codebase Document the data model implied by this prototype's mock data.

🔌 API_NEEDS.md for handoff

Every API endpoint the production version will need. For each: what it does, inputs, outputs. Prompt: @codebase List every place this prototype needs a real API call in production and describe each endpoint.

⚠️ KNOWN_ISSUES.md underrated

Things that are broken, stubbed, or intentionally incomplete. Saves awkward demo moments. Prompt: @codebase What's hardcoded, missing, or likely broken in this prototype?

🎬 Demo script

Not a code file — notes on how to walk through the prototype. What to show first, what to click, what to say. 10 minutes to write, makes every stakeholder demo 10x smoother.

Minimum viable artifact set For any prototype: PLANNING.md + TASKS.md + KNOWN_ISSUES.md. Enough for a dev to pick up where you left off and understand what's real vs. stubbed. Everything else is a bonus.
16 — SPEC CRAFT

Right level of detail

Too vague and the agent guesses wrong. Too detailed and you've done all the work yourself.

This is the craft of working with AI agents. There's a Goldilocks zone of specificity — finding it faster is what separates people who get great results from people who iterate forever.

❌ Too vague

"Build a dashboard page."

The agent invents everything. It produces something plausible but almost certainly not what you had in mind. You spend 45 minutes steering it back.

❌ Too specific

"Column 1 has a card at position x:0, y:0 with a blue (#3B82F6) background, 16px padding..."

You've written the code in English. It's slower than writing it yourself. The agent becomes a transcription service.

✅ Just right

"Build a dashboard page. Four metric cards at the top: total users, active this week, revenue this month, churn rate — use mock numbers. Below that, a line chart showing weekly signups over 8 weeks. Below that, a table of the 5 most recent signups: name, email, plan, signup date. Match the layout style of the other pages."

You've specified what is on the page and what data it shows. You've left how it looks to the agent. This is the zone.

The framework: specify What, not How

  • You specify: Which screens exist, what data appears, what actions a user can take, what the key interactions are.
  • Agent decides: Component structure, CSS details, file organization (within your rules), implementation approach.
  • You override when: The agent's choice is wrong for your audience, conflicts with your dev team's standards, or looks bad in the demo.

Signals you've written a good prompt

  • You can read it back and know exactly what you'd expect to see when it's done.
  • There's no ambiguity about what data or content should appear on screen.
  • You haven't specified colors, pixel values, or CSS properties (unless those genuinely matter).
  • You've told it what NOT to change — often as useful as what to do.
  • It references existing context: @PLANNING.md, existing component names, screen names from your file tree.

Iteration beats perfection

A good Cursor workflow is iterative, not waterfall. Write a solid-but-imperfect prompt, see what the agent builds, then correct specifically what's wrong. Three focused corrections beat one massive over-specified prompt every time.

# Round 1 — broad "Build the user settings page with sections for profile, notifications, and billing. Use mock data." # Round 2 — specific correction "The notifications section needs three toggle switches: email alerts, weekly digest, and product updates. Each needs a label and a short description below it." # Round 3 — scoped follow-up "Move billing to its own page at /settings/billing and add a link to it from the settings nav."
The PM advantage You're already trained to write acceptance criteria. Apply that muscle: "Given [context], when [action], then [outcome]." That structure translates almost directly into great Cursor prompts.
17 — FULL WORKFLOW

Full project workflow

The end-to-end playbook — from blank folder to handoff-ready prototype

Everything from the previous sections assembled into a single repeatable process. Use this as a checklist on your next project.

01

Start with a planning conversation

Open a new Chat. Describe your idea. Ask the AI to help you define screens, data, stack, and scope. Don't open a code file yet. Output: PLANNING.md + TASKS.md.

02

Create your folder and scaffold

New folder. Open in Cursor. Run Composer: Create a Next.js 14 app with TypeScript and Tailwind. Set up the folder structure from my PLANNING.md.

03

Add your three root files

Drop PLANNING.md, TASKS.md, and .cursorrules in the root before writing any real code. Every Composer session from here starts with @PLANNING.md @TASKS.md.

04

Build screen by screen

One Composer session per screen or feature. Be specific about what's on it. Review every diff. Commit to git after each working screen. Update TASKS.md as you go.

05

Wire it together

Once screens exist, add navigation. Link pages. Test the demo flow end-to-end. Ask: @codebase Review the navigation and make sure every page is reachable and linked correctly.

06

Polish the demo path

Add missing states (empty, loading, success). Make mock data tell a story — real-looking names, plausible numbers, a coherent narrative a user could follow.

07

Generate your handoff artifacts

@codebase Generate ARCHITECTURE.md, DATA_MODEL.md, API_NEEDS.md, and KNOWN_ISSUES.md for this prototype. Be specific and honest about what's stubbed or missing.
08

Record a walkthrough and ship

2–3 minute Loom. Walk through the demo path. Narrate what you built and what's intentionally out of scope. Share Loom + repo link. That's your handoff package.

🗓 Realistic time expectations
  • Simple 3-screen prototype — 2–4 hours from idea to demo-ready
  • Medium 6–8 screen prototype — Half a day to a full day
  • Complex multi-flow prototype — 2–3 days, especially if you want polish
  • Full handoff package — Add 1–2 hours to generate and review artifacts

These assume a clear PLANNING.md before you start building. The planning phase is not optional — it's what makes these timelines possible.

One last thing The workflow gets dramatically faster with practice. Your second prototype will be twice as fast as your first. By your fifth, you'll have your .cursorrules dialed in, your own PLANNING.md template, and a set of go-to prompts that reliably produce good results. That compounding is the real payoff.
